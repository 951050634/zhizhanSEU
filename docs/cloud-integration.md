# 火山云端接入准备说明

## 1. 当前阶段目标

当前处于无实物板阶段，云端工作重点不是宣称设备已接通，而是把后续真正接入火山方案所需的账号、控制台对象、许可证、消息字段和落库位置全部梳理清楚，让团队在板子到手后只需要补设备联调，不需要重新设计接入路径。

## 2. 赛题与官方约束

根据乐鑫物联网竞赛官网与官方说明，本项目在工程上需要满足以下基础要求：

- 主控芯片必须属于 `ESP32-S3`、`ESP32-C5` 或 `ESP32-P4`
- 推荐开发环境为 `ESP-IDF v5.5.2`
- 至少实现一种传感器融合
- 至少接入一个云端大模型
- 方案必须具备上行或下行交互

官方页面同时给出了云端合作方与示例资料，当前项目的主线云端方案固定采用火山方向：

- 乐鑫竞赛官网：[物联网赛题官方页面](https://www.espressif.com/zh-hans/ecosystem/education/competition/iot)
- 火山产品说明与权益入口：[火山引擎面向本赛题的说明页](https://www.volcengine.com/docs/6348/1913817?lang=zh)
- 语音对话智能体示例工程仓库：[ESP32-C5 语音智能体示例](https://github.com/espressif/esp-skainet/tree/master/examples/agent/volcengine_ai)

## 3. 为什么当前主线选火山硬件对话智能体

本项目的交互闭环是“终端语音采集 -> 云端理解 -> 网页大屏展示 -> 终端摘要反馈”。对这个链路而言，云端方案最关键的不是纯文本问答能力，而是对实时对话设备链路、设备许可管理和硬件接入友好度。因此当前优先沿用赛题官方已经给出的火山硬件对话智能体路线，而不是额外自造一套云端编排。

该选择在无板阶段的收益有三点：

- 它与赛题资源发放路径一致，后续申请支持和排障成本最低
- 它天然贴合语音交互终端，而不是只适合网页聊天框
- 它允许现在先冻结消息结构和控制台对象，再在板级联调时补齐真实链路

## 4. 无板阶段必须先拿到的控制台信息

在真正写设备端代码前，团队需要先完成以下控制台资产盘点。建议把实际值记录到内部私有清单中，不要直接提交到 Git。

- 火山账号已开通并具备访问竞赛权益的权限
- 已创建或确认可用的硬件对话智能体产品
- 已明确设备注册方式与设备名称命名规则
- 已确认可用许可证数量、试用额度和有效期
- 已绑定本项目准备使用的智能体或知识问答配置
- 已确认后续设备接入文档、示例仓库和 SDK 入口

建议采用如下命名方式管理设备：

- `zhizhan-demo-01`
- `zhizhan-demo-02`
- `zhizhan-final-01`

这样可以在演示设备、测试设备和最终展示设备之间保持清晰边界。

## 5. 本仓库建议冻结的配置项

当前阶段先不提交任何真实密钥，但应当明确后续代码中需要哪些配置位。建议后续把这些值分别落在本地私有配置文件或 CI Secret 中。

```json
{
  "provider": "volcengine-hardware-agent",
  "productKey": "",
  "deviceName": "zhizhan-demo-01",
  "agentName": "",
  "licenseNote": "",
  "region": "cn",
  "messageContractVersion": "2026-04-20",
  "notes": "真实密钥、票据和许可证信息不要入库"
}
```

如果后续网页端需要做纯模拟问答或联调占位，可额外准备一份只面向网页的 mock 配置，但不要把它与真实设备凭证混用。

## 6. 项目内推荐的数据流落点

当前工程建议采用如下分工：

- `docs/message-contract.md`：固定 `wake`、`select_exhibit`、`ask_question`、`cloud_result` 的消息结构
- `web/`：先消费模拟消息，后续切换为真实 `WebSocket` 输入
- `firmware/`：后续通过 `ESP-IDF v5.5.2` 与官方示例工程接近的方式接入火山链路

云端返回给本项目的中间数据结构必须固定包含以下字段：

- `intent`
- `title`
- `summary`
- `detail`
- `followup`

这些字段中，`summary` 与 `followup` 面向终端小屏，`detail` 面向网页大屏，`intent` 用于页面行为和日志调试。

## 7. 无板阶段的实际推进顺序

### 7.1 先完成网页与文档层

- 用 `web/` 页面先把 `cloud_result` 的结构消费起来
- 用 `docs/demo-script.md` 把比赛演示闭环写死
- 用 `docs/message-contract.md` 固定字段，避免后面重复改协议

### 7.2 再准备设备侧接入资料

- 从官方页面和示例仓库确认 `ESP32-C5` 的接入路径
- 梳理设备注册、许可证分配、智能体绑定的实际顺序
- 记录真实需要的配置项，但不把凭证写入仓库

### 7.3 板子到手后再做的动作

- 把设备从模拟事件源替换成真实 `wake`、`select_exhibit`、`ask_question`
- 按官方推荐环境补 `ESP-IDF v5.5.2` 工程
- 将网页中控从“模拟云端返回”切换到“真实云端返回”

## 8. 风险点与处理方式

### 8.1 风险点

- 当前没有板子，无法验证语音采集链路与许可证实际消耗
- 火山控制台字段名、接入凭证与 SDK 参数可能会因产品版本而变化
- 若网页阶段提前写死平台专有字段，后续设备接入会造成重复修改

### 8.2 处理方式

- 当前只冻结项目内部消息结构，不冻结任何未确认的平台私有字段
- 真实凭证统一放在私有本地配置，不进入 Git
- 真正设备接入时优先对照官方示例仓库，而不是以网页 mock 逻辑反推设备逻辑

## 9. 与 `.venv` 约束的关系

后续若需要写 Python 辅助脚本做消息回放、日志转换或批量生成演示数据，一律先在仓库根目录激活 `.venv` 再执行：

```powershell
Set-Location D:\Pass1on_W\Project\zhizhanSEU
.\.venv\Scripts\Activate.ps1
python script.py
```

这条规则是仓库基线，不允许退回系统 Python 混跑。
