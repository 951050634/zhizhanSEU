# 比赛演示闭环脚本

## 1. 文档定位

本脚本用于无板阶段的排练、录屏和网页中控联调。当前默认采用“网页模拟终端事件 + 模拟云端回包”的方式完成完整演示，不把任何板级能力包装成已验证事实。

## 2. 演示目标

在 3 至 4 分钟内让评委清楚看到四件事：

- 终端交互入口不是单一按钮，而是面向展厅导览的多模态入口
- 大屏网页不是普通宣传页，而是实时承接终端控制与云端结果的中控界面
- 云端大模型结果以结构化方式回流，能同时服务大屏与小屏
- 系统最终目标是“小屏交互、大屏联动、云端理解、低功耗待机”

## 3. 角色分工

- 讲解者：负责口播、切换网页控件和说明系统价值
- 操作手：当前阶段可由讲解者兼任，通过网页控制区触发模拟事件
- 大屏：展示 `web/` 中控页面
- 终端占位：当前阶段用口头说明“后续由 `ESP32-C5` 手持终端发出同名事件”

## 4. 建议演示流程

### 第一段：系统开场

口播重点：

“本方案面向展厅讲解场景，终端负责采集与唤醒，大屏负责展示，云端大模型负责理解与生成。现在因为实物板还未到位，我们先用同一套消息协议驱动网页中控，完成演示闭环验证。”

屏幕动作：

- 打开 `web/index.html`
- 展示主展示区、当前状态区、事件反馈区、演示控制区四个工作面
- 强调当前页面运行在模拟模式

### 第二段：唤醒与展项切换

操作顺序：

1. 点击 `唤醒终端`
2. 点击一个展项按钮，例如“芯片发展历程”

评委看到的结果：

- 当前状态由待机切换为激活
- 主展示区切到对应展项
- 事件反馈区出现 `wake` 与 `select_exhibit` 日志

讲解重点：

“未来这一动作对应真实终端的姿态唤醒与触摸选展项，现在先用完全一致的消息名做网页联调。”

### 第三段：发起问题并返回结果

操作顺序：

1. 在问题输入框填入一句问题，例如“这项技术为什么适合展厅讲解？”
2. 点击 `发起问答`

评委看到的结果：

- 事件反馈区出现 `ask_question`
- 页面短暂进入云端处理中状态
- 数百毫秒后出现 `cloud_result`
- 主展示区刷新标题、摘要、长文与追问建议

讲解重点：

“这里的云端结果不是自由文本，而是拆成 `intent`、`title`、`summary`、`detail`、`followup` 五类字段。未来终端只消费摘要，大屏消费长文，从而实现多屏协同。”

### 第四段：一键完整演示

操作顺序：

- 点击 `运行整段演示`

评委看到的结果：

- 页面自动依次执行唤醒、选展项、提问、返回结果
- 主展示区与状态区同步更新

讲解重点：

“这一步是为了在录屏、彩排和答辩现场快速复现完整闭环，避免手工操作不稳定。”

### 第五段：说明板子到手后的衔接

口播重点：

“后续板子到位后，我们不重写页面，只把模拟事件源替换为真实 `WebSocket` 输入；同样地，固件只需要按今天已经冻结的消息结构发送 `wake`、`select_exhibit`、`ask_question` 和 `cloud_result` 即可接入现有中控。”

## 5. 演示时建议使用的问题

- 这项技术为什么适合展厅导览？
- 多屏联动相比单屏讲解有什么优势？
- 终端和大屏各自承担什么角色？
- 如果观众继续追问，系统如何保持上下文？

## 6. 录屏与答辩准备清单

- 大屏浏览器提前打开并进入全屏
- 问题输入框预填一条稳定问题
- 事件区日志确保可见，便于证明链路确实发生
- 如需答辩录屏，优先使用“一键完整演示”
- 准备一页口头说明，明确当前仍是无板阶段，板级验证待后续完成

## 7. 失败预案

- 如果自动演示失效，改用单步按钮逐项触发
- 如果问题输入框内容异常，使用页面内置追问按钮触发
- 如果大屏需要更强可解释性，重点展示右侧消息预览与字段结构

## 8. 当前阶段输出标准

满足以下条件即可认为无板阶段演示闭环已经成立：

- 页面能稳定展示四个工作区
- `wake`、`select_exhibit`、`ask_question`、`cloud_result` 四类事件都能被触发
- 主展示区能消费结构化结果
- 讲解词能明确区分“已实现的页面闭环”和“待硬件验证的设备链路”
